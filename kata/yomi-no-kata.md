# Yomi no Kata

## Why this is a relevant kata

This kata formalizes the monitoring side of Cyberjutsu. It is relevant because the repository repeatedly teaches that reading logs and live system activity is not a support skill but a central martial skill. The kata transforms passive log viewing into an intentional method for seeing hostile movement as it begins.

### Criteria from [kata-common-principles](../references/kata-common-principles.md)

- `Clear pedagogical purpose`: teach disciplined monitoring and rapid extraction of useful signal from live logs.
- `Fixed structure with repeatable standard`: prepare the screen, launch monitoring, trigger an event, identify the source.
- `Technical coherence`: built directly from `Yomi Waza` exercises already taught in the repository.
- `Embodied principles`: trains attention, discrimination, timing, and calm under ongoing activity.
- `Link to application`: applies to scan detection, connection awareness, and incident triage.
- `Progressive difficulty`: can move from single-event reading to filtering under noisy conditions.
- `Precision of intent`: the practitioner must look for source IP, destination port, and type of activity.
- `Rhythm and maai`: preparation, stillness, event appearance, recognition, and verbal confirmation.
- `Control and safety`: uses normal monitoring without destructive action.
- `Ethical and cultural frame`: teaches patient observation before reaction.

## Type

Paired kata.

## Roles

- `Tori`: the monitor
- `Uke`: the activity generator

## Objective

Tori detects and explains the activity generated by uke from live system logs.

## Sequence

1. Both practitioners perform `rei`.
2. Tori prepares a split practice view.
```bash
screen
```
3. Tori opens a monitoring pane and starts log reading.
```bash
tail -f /var/log/syslog
```
4. Uke performs one bounded action against tori, for example a port scan.
```bash
nmap $TORI_IP_ADDRESS
```
5. Tori reads the displayed lines and states:
   - the source IP
   - the destination port or ports
   - the likely category of action
6. Tori narrows the signal with a filter when needed.
```bash
tail -f /var/log/syslog | grep -i $CATCH_STRING
```
7. Uke repeats the action.
8. Tori confirms that the filtered view isolates the relevant trace.
9. Roles switch.

## Success criteria

- Tori maintains monitoring before the action starts.
- Tori identifies the source of the event correctly.
- Tori identifies a useful filter string.
- Tori explains the event without guessing beyond the observed evidence.

## Progressive variants

- Beginner: single scan from one source.
- Intermediate: repeated scans with changing ports.
- Advanced: mixed benign and hostile traffic where tori must isolate only the relevant signal.
